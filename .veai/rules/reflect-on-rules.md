# reflect-on-rules

**Оценка необходимости новых правил после крупных задач.**

## Когда применять

- После завершения крупной задачи (рефакторинг, добавление нового API, реализация фичи)
- При обнаружении повторяющихся паттернов, которые стоит зафиксировать
- При возникновении ошибок, которых можно было бы избежать с правилом

## Правила

### 1. Оценивать после каждой крупной задачи

По завершении работы спросить себя:

- Были ли паттерны, которые повторялись несколько раз?
- Возникли ли неоднозначности, которые стоило бы зафиксировать?
- Научились ли мы чему-то новому, что стоит сохранить?
- Есть ли ошибки, которые можно было бы предотвратить правилом?

### 2. Предлагать пользователю

Если ответ на любой из вопросов — «да», предложить пользователю зафиксировать новые правила:

> «Я заметил, что [описание паттерна/проблемы]. Стоит ли добавить правило в `.veai/rules/` или обновить существующее?»

### 3. Не коммитить артефакты ревью

- Не коммитить файлы анализа, DEEP-REVIEW.md и подобные артефакты без явного запроса
- Не добавлять документацию к внутренним/private элементам без запроса
- Не расширять публичный API без запроса
